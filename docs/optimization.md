# Анализ и оптимизация запроса «Топ-5 за последний месяц»

## Анализ текущего запроса

Запрос из п. 2.3.1:
- Рекурсивный CTE `category_root` — обход дерева категорий до корня (1-й уровень).
- Фильтр по заказам за последний календарный месяц (`orders.created_at`).
- JOIN: orders → order_items → nomenclature → category_root.
- Группировка по товару и корневой категории, сумма количества, сортировка, LIMIT 5.

При тысячах заказов в день за месяц накапливаются десятки тысяч заказов и сотни тысяч строк в `order_items`. Каждый запуск запроса сканирует все эти строки и пересчитывает рекурсию по категориям.

---

## Оптимизация запроса

### 1. Индексы

Уже есть по FK (`order_id`, `nomenclature_id`, `category_id`, `parent_id`). Добавить:

- **`orders (created_at)`** или **`orders (created_at, id)`** — отбор за последний месяц без полного скана.
- **`order_items (order_id)`** — уже даёт FK-индекс; при необходимости составной **`(order_id, nomenclature_id)`** для группировки/агрегации по товару в рамках заказа.

### 2. Убрать рекурсию из отчёта: денормализация «корень категории»

Рекурсивный CTE по категориям выполняется при каждом вызове. Дерево категорий обычно небольшое, но его можно не пересчитывать в отчёте.

**Вариант:** в таблице `categories` добавить поле **`root_id`** (FK на `categories.id`) — id категории 1-го уровня. Заполнять при вставке/обновлении категории (триггер или код приложения). В отчёте один JOIN с `categories root ON root.id = n.category.root_id`, без CTE.

Плюсы: запрос проще, нет рекурсии. Минусы: дублирование данных, нужно поддерживать целостность при изменении дерева.

### 3. Материализованное представление

Если отчёт не нужен в реальном времени, создать **материализованное представление** по тому же запросу и обновлять по расписанию (cron, например раз в час или ночью):

```sql
CREATE MATERIALIZED VIEW mv_top5_products_last_month AS
<тело запроса 2.3.1>
WITH DATA;

CREATE UNIQUE INDEX ON mv_top5_products_last_month (product_name, category_level1);
```

Обновление: `REFRESH MATERIALIZED VIEW mv_top5_products_last_month;`

Чтение отчёта — быстрый скан маленькой таблицы без JOIN по заказам.

## Оптимизация схемы данных при росте нагрузки

### 4. Партиционирование по времени

При большом объёме заказов:

- **`orders`** — партиционировать по `created_at` (например по месяцу: `PARTITION BY RANGE (created_at)`).
- **`order_items`** — партиционировать по `order_id` не всегда удобно; альтернатива — партиционирование по диапазону даты заказа, если хранить дату и в `order_items` (дублирование), или партиционировать только `orders`, а `order_items` джойнить по `order_id` (партиционированный join в PostgreSQL отсекает партиции по условию на `orders.created_at`).

В результате запрос «за последний месяц» обращается в основном к одной партиции `orders` и связанным строкам `order_items`.

### 5. Агрегатная таблица (сводка по продажам)

Отдельная таблица дневной/месячной агрегации:

- **`sales_aggregate`**: `(date`, `nomenclature_id`, `root_category_id`, `quantity_sold)` — сумма проданных штук по товару и корневой категории за день.

Заполнение: ночной batch или триггеры при проведении заказа (сложнее). Запрос «топ-5 за последний месяц» — суммирование по `sales_aggregate` за последние 30 дней, без обращения к `order_items` и к полному объёму заказов.

Плюсы: быстрые отчёты, предсказуемая нагрузка. Минусы: ETL или логика согласованности, задержка данных при batch-подходе.

### 6. Покрывающий индекс (если без денормализации)

Если оставить один общий запрос к `orders`/`order_items`, но добавить индекс, который позволит не ходить в таблицу за лишними полями, можно рассмотреть составной индекс под план запроса, например по `orders`: `(created_at, id)` и по `order_items`: `(order_id, nomenclature_id, quantity)`, чтобы агрегация по товару и количеству по возможности выполнялась по индексу. Эффект зависит от плана (EXPLAIN ANALYZE), не всегда планировщик выберет index-only scan.

## Рекомендуемый порядок внедрения

1. **Индекс по `orders.created_at`** — минимальное изменение, сразу снижает объём сканирования.
2. **Материализованное представление** для отчёта — если допустима задержка (минуты/часы).
3. **Поле `root_id` в категориях** — убирает рекурсию из всех отчётов по «категории 1-го уровня».
4. При дальнейшем росте — **партиционирование `orders`** по месяцу.
5. При необходимости максимально быстрых отчётов — **таблица агрегатов** по дням и пересчёт топа за месяц по ней.
